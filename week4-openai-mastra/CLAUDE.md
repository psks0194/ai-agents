> The name is misleading — this directory is the **OpenAI Agents SDK** week. The Mastra week lives separately at `../week4-mastra/`.

## What this is

The canonical agent (typed output, tool loop, structured research-note) built on the **OpenAI Agents SDK** — a small surface where `Agent` is the declaration and `Runner` is the engine. Day 1 ends with a **three-way bake-off** against Pydantic AI and LangGraph (extending Week 3's two-way comparison). Day 2 covers the SDK's signature features: **handoffs** (model-driven control transfer) and **guardrails** (input/output tripwires).

## Commands

From inside `week4-openai-mastra/`:

```bash
uv sync
uv run python -m week4_openai_mastra.first_agent              # typed output, no tools
uv run python -m week4_openai_mastra.tool_agent               # Runner runs the tool loop
uv run python -m week4_openai_mastra.research_note_oai        # OpenAI version of the Week 3 task
uv run python -m week4_openai_mastra.compare_three            # Pydantic AI vs LangGraph vs OpenAI SDK
uv run python -m week4_openai_mastra.handoffs                 # triage → billing/technical specialists
uv run python -m week4_openai_mastra.guardrails               # input + output tripwires
```

`compare_three.py` calls Anthropic (for Pydantic AI + LangGraph) **and** OpenAI (for the OpenAI SDK) — needs both keys in the repo-root `.env`. A low balance on either provider fails only that provider's runs, not the others.

## The cross-week import (`compare_three.py`)

`compare_three.py` extends Week 3's two-way comparison. To do that, it reaches into `../week3-langgraph-pydantic-ai/src/` via a **`sys.path` hack at lines 14–18** to import Week 3's `research_note_pai` and `research_note_lg`.

That makes Week 3's *code* importable, but **not** its *dependencies* — so `pyproject.toml` here re-declares `pydantic-ai`, `langgraph`, and `langchain-anthropic`. If Week 3's signatures change, this script breaks; if you bump those libraries, bump both `pyproject.toml`s. A uv workspace would replace the path hack — until then, treat this dependency as load-bearing.

## Gotchas

- **Global key, not per-agent.** The SDK has no `api_key=` on `Agent`. Every module calls `set_default_openai_key(settings.openai_api_key)` once at import — keep that pattern when adding new entry points.
- **Tools take no `ctx`.** Unlike Pydantic AI's `@agent.tool` (which requires `ctx: RunContext[Deps]` first), `@function_tool` decorates a plain function. Docstring + type hints become the schema.
- **Handoffs are declared after both ends exist.** In `handoffs.py`, `billing_agent.handoffs = [triage_agent]` is wired *after* `triage_agent` is defined, to avoid a circular reference at construction. Preserve that ordering when adding return handoffs.
- **Guardrail tripwires raise.** `tripwire_triggered=True` halts the run with `InputGuardrailTripwireTriggered` / `OutputGuardrailTripwireTriggered`. Callers catch those and show a safe fallback — don't refactor that into normal returns.
- **`decision_matrix_2026.html`** is a hand-written visual artifact (the four-framework decision matrix), not generated by code. Don't expect to regenerate it from a script.
